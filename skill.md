# 功能开发与持续集成工作流
**技能标识**：feature-dev-ci-workflow

## 一、技能描述
本技能规范研发人员从功能分支创建、本地编码自测、代码提交评审，到CI自动化构建、制品打包上传、测试环境自动部署的全流程研发规范。适用于**GitHub代码仓库+CI流水线+企业制品库**标准化团队研发场景。

## 二、触发条件
1. 新增业务功能、线上缺陷修复开发需求
2. 需求文档、用户故事、开发任务已明确落地

## 三、前置条件
1. 代码仓库托管至GitHub，线上主干默认分支为`main`
2. CI流水线已完成配置（GitHub Actions/Jenkins），内置能力：
   - 代码格式规范校验
   - 静态代码漏洞分析
   - 单元测试执行+覆盖率统计
   - 项目打包构建、制品推送上传
3. 测试环境已打通自动化部署，监听主干分支CI成功事件自动发布
4. 开发本地环境配齐项目SDK、运行依赖、代码格式化工具、单元测试组件

## 四、输入内容
1. 功能需求文档 / Bug缺陷描述
2. 项目技术设计方案（可选）

## 五、最终输出
1. 成功合并至`main`主干分支的完整业务代码
2. 评审通过、全量CI检测无异常的正式代码版本
3. 上传至制品库标准化部署包（示例：user-service-v1.2.3.jar）
4. 自动部署完成、可正常联调测试的线上测试环境服务

## 六、标准工作流程
### 1. 拉取主干代码，创建独立功能分支
```bash
git checkout main
git pull origin main
git checkout -b feat/功能简短名称
```

### 2. 本地编码开发+单元测试编写
1. 按照团队编码规范完成业务逻辑开发
2. 编写单元测试，覆盖三大场景
   - 正常业务流程校验
   - 空值、重复数据、格式错误等异常边界
   - 业务状态流转、权限判断规则校验
3. 使用Mock模拟数据库、第三方接口、文件服务等外部依赖
4. 本地完成全量测试，保证所有用例运行成功

### 3. 本地代码质量自检
1. 执行代码格式化工具统一代码风格
2. 本地运行静态代码检查，提前修复代码隐患
3. 提交代码并推送远程分支
```bash
git add .
git commit -m "feat: 完成XX功能开发及单元测试"
git push -u origin feat/功能简短名称
```

### 4. 提交GitHub PR合并请求
1. 发起功能分支向`main`主干分支的Pull Request
2. PR填写规范：写明开发内容、测试方案、重点注意事项

### 5. PR触发CI自动化流水线校验
1. 自动拉取依赖、编译构建项目
2. 执行代码风格检测、静态漏洞扫描，失败直接拦截
3. 批量执行单元测试，统计代码覆盖率，不达标禁止通过
4. 全量检测通过后，打包生成正式部署制品
5. 自动将制品推送至Nexus/Artifactory制品库归档

### 6. 团队代码评审与分支合并
1. 评审重点：业务逻辑、测试用例完整性、异常处理、代码冗余
2. 整改完毕通过评审后，合并分支至main主干
3. 推荐合并方式：Squash合并 / Rebase合并

### 7. 主干触发正式CI，自动部署测试环境
1. 代码合入main分支后触发正式生产级CI流水线
2. 流水线构建最新版本制品
3. 自动化下发部署脚本，完成测试环境服务更新
4. 部署完成发送通知，进入集成联调测试阶段

## 七、强制合规检查点
1. 单元测试严禁连接真实数据库与线上第三方服务，必须使用Mock隔离
2. PR所有CI校验项全部通过，才可进行代码合并
3. 项目代码覆盖率满足团队最低标准（建议≥60%行覆盖率）
4. 所有部署制品版本号唯一，与代码提交记录一一绑定
