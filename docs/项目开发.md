## 1. 项目概述

### 1.1 项目背景

随着大语言模型（LLM）技术的快速发展，传统的即时通讯系统正面临深刻变革。本项目旨在构建一个 **AI-Native（AI原生）的即时通讯系统**，将 AI 能力深度融入通讯的每一个环节，使用户能够在聊天的任何场景中无缝调用 AI 辅助。

### 1.2 项目目标

- 构建一个支持 **实时通讯** 的聊天平台（私聊、群聊、频道）
- 将 **DeepSeek LLM** 深度集成到聊天体验中，提供 AI 助手功能
- 实现基于 **ReAct + Plan-and-Execute** 的 AI Agent，具备长期记忆和知识检索能力
- 支持 **RAG（检索增强生成）**，允许用户上传文档构建私有知识库
- 提供 **OAuth2 认证**（GitHub、Google）以及 JWT 会话管理
- 确保系统的 **可扩展性** 与 **可维护性**

### 1.3 核心功能

| 功能模块 | 描述 |
|---------|------|
| 即时通讯 | 支持私聊、群聊、频道，支持文本、图片、文件、语音、视频消息 |
| 实时推送 | 基于 Socket.io 的 WebSocket 实时消息推送 |
| AI Agent | DeepSeek V3（快速响应）+ DeepSeek R1（深度推理）双模型 |
| Agent 推理模式 | ReAct（快速响应）、Planner（计划执行）、Reasoner（深度推理） |
| Agent 记忆 | 短期记忆（Redis，30min TTL）+ 长期记忆（Prisma + pgvector） |
| 知识库 RAG | 支持文本和文档上传，向量化存储，语义检索 |
| OAuth 登录 | 支持 GitHub 和 Google 第三方登录 |
| JWT 会话 | Access Token + Refresh Token 双令牌机制 |
| 文件存储 | MinIO（S3 兼容）对象存储 |
| 通知系统 | 未读消息、好友请求、系统通知推送 |
| 审计日志 | 记录关键操作的审计日志 |

### 2.2 架构分层说明

| 层级 | 说明 | 技术选型 |
|------|------|---------|
| **客户端层** | 用户交互界面 | React 18 + TypeScript + Vite |
| **网关层** | 流量接入、负载均衡 | Nginx |
| **应用层** | 业务逻辑处理 | NestJS + TypeScript |
| **服务层** | 数据持久化与缓存 | PostgreSQL + Redis + MinIO |
| **AI 层** | 大语言模型推理 | DeepSeek V3 / DeepSeek R1 |

### 2.3 数据流向

#### 常规聊天消息流

```
客户端 A → HTTP/WS → Nginx → NestJS API → PostgreSQL → Socket.io → 客户端 B
```

#### AI Agent 消息流

```
客户端 → HTTP/WS → Nginx → AgentController → AgentOrchestrator
                                            ├── IntentClassifier（意图分类）
                                            ├── [ReAct Engine] ──┬→ ToolRegistry
                                            │                    └→ LLM（DeepSeek V3）
                                            └── [PlanExecutor] ──┬→ ToolRegistry
                                                                   └→ LLM（DeepSeek R1）
                                                           ↓
                                                    RAG Engine（知识检索）
                                                           ↓
                                                    Memory Service（记忆管理）
                                                           ↓
                                                       响应
```

### 3.2 Monorepo 配置

本项目使用 **pnpm workspaces** 管理 monorepo，所有包管理操作均在根目录执行：

```
根目录/
├── apps/
│   ├── api/          → ai-native-chat-api
│   └── web/          → ai-native-chat-web
├── libs/             → 共享库（预留）
├── docker/           → Docker 配置文件
├── docs/             → 项目文档
├── .env.example      → 环境变量模板
├── pnpm-workspace.yaml
├── package.json      → 根 workspace 配置
└── README.md
```

### 7.2 意图分类

| 意图类别 | 说明 | 路由策略 |
|---------|------|---------|

### 7.3 ReAct Engine

ReAct（Reasoning + Acting）是一种将推理与行动结合的 Agent 范式。模型在每一步生成**思考（Thought）**、**行动（Action）** 和**观察（Observation）**，循环执行直到得出最终答案。

```
循环 (最多 10 步):
  1. Thought: 分析当前状态，决定下一步行动
  2. Action: 调用工具或查询知识
  3. Observation: 获取行动结果
  4. 判断是否完成，否则继续循环
```

**支持的内置工具：**

| 工具名 | 描述 | 输入 |
|--------|------|------|
| `web_search` | 搜索互联网 | 查询字符串 |
| `calculate` | 数学计算 | 数学表达式 |
| `get_time` | 获取当前时间 | — |
| `get_weather` | 查询天气 | 城市名 |
| `search_knowledge` | 搜索私有知识库 | 查询字符串 |
| `memory_recall` | 回忆相关记忆 | 关键词 |

### 7.4 Plan-and-Execute

Plan-and-Execute 是一种分层规划方法：

1. **规划阶段（Plan）**：使用 LLM 分析任务，生成多步骤执行计划
2. **执行阶段（Execute）**：按顺序执行每个步骤
3. **反思阶段（Reflect）**：评估执行结果，如有需要重新规划

```
用户问题 → 规划器(LLM) → 执行计划
                           ↓
                    [Step 1] → [Step 2] → [Step 3] → ...
                           ↓
                    结果聚合 → 最终回答
```

### 7.5 记忆系统

**短期记忆（Redis）**

- TTL：30 分钟
- 存储内容：当前会话的上下文消息摘要
- 访问频率：高
- 淘汰策略：基于时间 + 重要性评分

**长期记忆（PostgreSQL + pgvector）**

- 持久化存储
- 存储内容：用户偏好、跨会话关键信息、AI 总结的要点
- 向量检索：基于语义相似度召回
- 重要性管理：基于交互频率和质量评分

**记忆压缩**：当短期记忆超过阈值时，调用 LLM 生成摘要并选择性存入长期记忆。

### 7.6 RAG 流程

```
用户查询
    │
    ▼
Embedding 模型 → 查询向量
    │
    ▼
pgvector 相似度检索 (top-k=5)
    │
    ▼
上下文组装 (用户查询 + 检索片段)
    │
    ▼
LLM 生成最终回答
    │
    ▼
回答 + 引用来源
```

```typescript
const client = axios.create({
  baseURL: '/api',
  timeout: 30000,

// 请求拦截器：自动附加 Token
client.interceptors.request.use((config) => {
  const token = authStore.getState().accessToken
  if (token) {
    config.headers.Authorization = `Bearer ${token}`
  }
  return config
})

// 响应拦截器：处理 401，自动刷新 Token
client.interceptors.response.use(
  (response) => response,
  async (error) => {
    if (error.response?.status === 401) {
      try {
        await authStore.getState().refreshToken()
        return client.request(error.config)
      } catch {
        authStore.getState().logout()
        window.location.href = '/login'
      }
    }
    return Promise.reject(error)
  }
)
```

### 9.2 DTO 校验

所有 DTO 使用 `class-validator` 进行校验：

```typescript

export class CreateSessionDto {
  @IsEnum(SessionType)
  type: SessionType

  @IsArray()
  @IsUUID('4', { each: true })
  memberIds: string[]

  @IsOptional()
  @IsString()
  @MaxLength(100)
  name?: string
}
```

```
                ↓
         RateLimitGuard (可选)
```

### 9.4 WebSocket 网关

```typescript
@WebSocketGateway('/chat', { namespace: '/chat' })
export class ChatGateway {
  @SubscribeMessage('message')
  async handleMessage(
    @MessageBody() data: SendMessageDto,
    @ConnectedSocket() client: Socket,
  ) {
    const user = await this.authService.validateToken(
      client.handshake.auth.token,
    )
    const message = await this.chatService.sendMessage(user.id, data)
    // 广播到会话房间
    this.server.to(`session:${data.sessionId}`).emit('message', message)
  }
}
```

### 13.1 Git 规范


类型（type）：`feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`

### 13.2 代码风格

#### TypeScript

- 使用 `interface` 定义对象类型，`type` 定义联合/交叉类型
- 优先使用 `const`，避免使用 `var`
- 使用 `unknown` 替代 `any`，必要时使用类型断言
- 导出类型定义以便复用

```typescript
// Good
export interface UserProfile {
  id: string
  username: string
  email: string
  avatarUrl?: string
}

export type AgentMode = 'REACT' | 'PLANNER' | 'REASONER'

// Bad
const user: any = {}
const user = {} as any
```

#### NestJS

- 控制器只处理请求/响应，业务逻辑放在 Service
- Service 使用依赖注入，避免直接 `new`
- 使用 DTO 进行数据校验和转换
- 守卫用于认证/授权，切面用于日志/监控

#### React

- 组件使用函数组件 + Hooks
- 状态提升到必要的最小层级
- 自定义 Hook 封装可复用逻辑
- 列表渲染使用 `key`，优先使用 `id` 而非索引

### 13.3 安全性规范

| 规范 | 说明 |
|------|------|
| **密码存储** | 使用 bcryptjs 哈希，不得明文存储 |
| **Token 传输** | 仅通过 HTTPS 传输，JWT 不得包含敏感信息 |
| **SQL 注入** | 使用 Prisma ORM 参数化查询，不使用字符串拼接 |
| **XSS** | 前端对用户输入进行转义，使用 React 的自动转义 |
| **CSRF** | 使用 SameSite Cookie + CORS 白名单 |
| **速率限制** | 登录接口实施限流（5次/分钟） |
| **敏感数据** | API 响应中排除密码哈希、Token 等敏感字段 |
| **审计日志** | 记录登录、敏感操作等关键事件 |

### 13.4 性能优化规范

| 场景 | 优化策略 |
|------|---------|
| **数据库查询** | 使用 `select` 明确字段，避免 `include` 过度嵌套 |
| **列表分页** | 消息列表默认 limit=50，最大 200 |
| **缓存策略** | Redis 缓存热点数据，设置合理 TTL |
| **向量检索** | pgvector 使用 HNSW 索引，top-k 限制 10 |
| **前端渲染** | 大列表使用虚拟滚动（预留） |
| **LLM 调用** | 流式响应减少等待感，设置超时 |
| **WebSocket** | 消息分页加载，房间人数限制 500 |

### 13.5 错误处理规范

```typescript
// 后端：统一异常过滤器
@Catch()
export class AllExceptionsFilter implements ExceptionFilter {
  catch(exception: unknown, host: ArgumentsHost) {
    const ctx = host.switchToHttp()
    const response = ctx.getResponse()
    const request = ctx.getRequest()

    const status = this.getStatus(exception)
    const message = this.getMessage(exception)

    response.status(status).json({
      statusCode: status,
      timestamp: new Date().toISOString(),
      path: request.url,
      message,
    })
  }
}

// 前端：统一错误处理
client.interceptors.response.use(
  (response) => response,
  (error) => {
    const { response } = error
    if (response?.status === 400) {
      toast.error(response.data.message || '参数错误')
    } else if (response?.status === 403) {
      toast.error('无权限操作')
    } else if (response?.status === 500) {
      toast.error('服务器错误，请稍后重试')
    }
    return Promise.reject(error)
  }
)
```
