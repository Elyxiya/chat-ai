## 1. Git 规范

### 1.1 分支策略

采用 Git Flow 变体，长期分支为 `main` 和 `develop`：

| 分支 | 用途 | 保护 |
|------|------|------|
| `main` | 生产就绪代码 | ✅ 需 PR 合入 + CI 通过 |
| `develop` | 开发集成分支 | ✅ 需 PR 合入 |
| `feature/<id>-<desc>` | 功能开发 | 从 `develop` 切出，合回 `develop` |
| `fix/<id>-<desc>` | Bug 修复 | 从 `develop` 或 `main` 切出 |
| `release/<version>` | 发布准备 | 从 `develop` 切出，合入 `main` 和 `develop` |

### 1.2 分支命名

```
<type>/<issue-id>-<short-kebab-desc>
```

示例：
- `feature/123-user-auth`
- `fix/456-token-refresh`
- `refactor/agent-orchestrator`
- `docs/update-api-spec`

类型前缀：`feature` / `fix` / `refactor` / `docs` / `test` / `chore`

### 1.3 提交信息

采用 Conventional Commits 格式：

```
<type>(<scope>): <subject>

<body>

<footer>
```

**type（必填）：**

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(search): 全局消息搜索 — 后端分组 + 原始 SQL` |
| `fix` | Bug 修复 | `fix(editor): 修复 Tiptap 工具栏在深色模式下的样式` |
| `refactor` | 重构 | `refactor(agent): AgentOrchestrator 意图路由解耦` |
| `docs` | 文档 | `docs: 更新 API 接口规范文档` |
| `test` | 测试 | `test: 新增 ChatModule 单元测试 15 个` |
| `chore` | 构建/工具 | `chore: pnpm v10 → v11 迁移` |

**scope（可选）：** 影响的模块名，如 `auth`、`chat`、`agent`、`knowledge`、`editor`、`ci`

**subject（必填）：** 中文描述 + 英文说明，简洁明了
```
feat(search): 智能消息搜索增强 — 后端分组 + 原始 SQL
fix: 消除 Prisma $queryRaw 的 text = uuid 类型不匹配
```

### 1.4 Pull Request 流程

1. 从 `develop` 创建 `feature/xxx` 分支
2. 开发完成后提交 PR 到 `develop`
3. PR 标题与 commit 主题一致
4. PR 描述包含：
   - 改动摘要
   - 关联 Issue（如 `Closes #123`）
   - 测试说明
   - 截图（UI 改动时）
5. 至少 1 人 Code Review 通过后合入
6. 合入方式：**Squash merge**（保持历史整洁）

## 2. TypeScript 编码规范

### 2.1 类型定义

```typescript
// ✅ interface 定义对象类型（可扩展、可合并）
export interface UserProfile {
  id: string;
  username: string;
  email: string;
  avatarUrl?: string;
}

// ✅ type 定义联合/交叉类型、工具类型
export type AgentMode = 'react' | 'planner' | 'reasoner';
export type ApiResponse<T> = { code: number; data?: T; message: string };

// ❌ 禁止使用 any，使用 unknown 替代（必要时用 as 断言）
const data: unknown = await response.json();
const user = data as UserProfile;
```

### 2.2 命名约定

| 类别 | 约定 | 示例 |
|------|------|------|
| 文件/目录 | kebab-case | `chat.service.ts`、`tool-registry.ts` |
| 类/组件/接口 | PascalCase | `class ChatService`、`interface ChatMessage` |
| 变量/函数/方法 | camelCase | `const accessToken`、`function getUser()` |
| 常量 | UPPER_SNAKE_CASE | `BCRYPT_ROUNDS = 12`、`SHORT_TERM_TTL` |
| 类型/接口前缀 | 无前缀 | `interface UserConfig` 而非 `IUserConfig` |
| Prisma 模型 | PascalCase + 单数 | `model ChatSession` |

### 2.3 严格模式要求

- 后端：`strictNullChecks: true` + `noImplicitAny: true`
- 前端：`strict: true`（全开）
- 函数参数和返回值优先显式声明类型
- `any` 类型由 ESLint 告警，必须附加注释说明原因

```typescript
// ✅ 正确
async function getUser(id: string): Promise<User | null> {
  return this.prisma.user.findUnique({ where: { id } });
}

// ❌ 避免
async function getUser(id: string) {
  return this.prisma.user.findUnique({ where: { id } });
}
```

## 3. 后端开发规范（NestJS）

### 3.1 模块结构

每个功能模块遵循统一目录结构：

```
module-name/
├── module-name.module.ts      # 模块定义（@Module 装饰器）
├── module-name.controller.ts  # 路由控制器（含 Swagger 装饰器）
├── module-name.service.ts     # 业务逻辑层
├── dto/
│   ├── create.dto.ts          # 创建/写入 DTO
│   ├── update.dto.ts          # 更新 DTO（可选）
│   └── query.dto.ts           # 查询参数 DTO
├── guards/                    # 守卫（可选）
│   └── *.guard.ts
├── strategies/                # Passport 策略（可选）
├── decorators/                # 自定义装饰器（可选）
├── interfaces/                # 类型/接口（可选）
└── *.spec.ts                  # 单元测试
```

**依赖注入规则：**
- Controller 只处理请求/响应，不写业务逻辑
- Service 包含业务逻辑，通过 constructor 注入依赖
- Module 集中注册 providers 和 exports

```typescript
@Module({
  imports: [LLMModule],
  controllers: [KnowledgeController],
  providers: [KnowledgeService, FileParserService, RagEngine],
  exports: [KnowledgeService, RagEngine],
})
export class KnowledgeModule {}
```

### 3.2 API 设计

**路由版本化：**
```typescript
@Controller({ path: 'chat', version: '1' })
// 实际路径: /api/v1/chat/sessions
```

**RESTful 命名约定：**

| 方法 | 路径 | 用途 |
|------|------|------|
| `GET` | `/resource` | 列表查询 |
| `GET` | `/resource/:id` | 获取单个 |
| `POST` | `/resource` | 创建 |
| `PATCH` | `/resource/:id` | 部分更新 |
| `DELETE` | `/resource/:id` | 删除 |
| `POST` | `/resource/:id/action` | 自定义操作 |

**响应格式：** 统一使用 `success()` 包装
```typescript
// 成功
{ code: 0, message: "Success", data: {...}, timestamp: "2025-01-15T10:30:00Z" }

// 错误（由全局异常过滤器处理）
{ statusCode: 400, message: "参数错误", path: "/api/v1/chat/sessions", timestamp: "..." }
```

**认证模式：**
```typescript
@ApiTags('Chat')
@ApiBearerAuth()
@UseGuards(JwtAuthGuard)  // 类级别：全部接口需认证
@Controller({ path: 'chat', version: '1' })
export class ChatController {
  @Get('sessions')
  async getSessions(@CurrentUser('id') userId: string) {
    return success(await this.chatService.getUserSessions(userId));
  }
}
```

**Swagger 文档：**
- 每个端点标注 `@ApiOperation({ summary: '...' })`
- DTO 字段标注 `@ApiProperty({ description: '...' })`

### 3.3 数据校验

```typescript
// dto/chat.dto.ts
import { IsEnum, IsUUID, IsOptional, MaxLength } from 'class-validator';

export class CreateSessionDto {
```

### 3.4 Prisma 数据模型约定

```prisma
// ✅ 命名规范
model ChatSession {
  id         String             @id @default(uuid())      // UUID 主键
  type       String             @map("type")              // camelCase 字段
  name       String?            @map("name") @db.VarChar(200)
  ownerId    String             @map("owner_id")          // FK: 关联表_id
  createdAt  DateTime           @default(now()) @map("created_at")
  updatedAt  DateTime           @updatedAt @map("updated_at")

  owner      User               @relation(fields: [ownerId], references: [id])
  members    ChatSessionMember[]

  @@index([ownerId])                                     // 查询索引
  @@map("chat_sessions")                                 // 蛇形表名
}
```

**规则：**
- 模型名：PascalCase 单数
- 表名：`@@map("snake_case_plural")`
- 字段：camelCase 代码名 → `@map("snake_case")` 列名
- 主键：`id String @id @default(uuid())`
- 时间戳：`createdAt` / `updatedAt` 统一命名
- 外键：`<关联模型>Id` + `@map("<关联表>_id")`
- 级联：`onDelete: Cascade` 从属关系；`onDelete: SetNull` 可选引用

**子组件**放在对应页面的组件目录或共享的 `src/components/`。

### 4.2 组件命名与导出

```typescript
// ✅ 使用默认导出，文件名 = 组件名
interface MessageBubbleProps {
  message: ChatMessage;
  isOwn: boolean;
  onReply?: () => void;

export default function MessageBubble({ message, isOwn, onReply }: MessageBubbleProps) {
```

**规则：**
- 函数组件 + Hooks，不使用 class 组件

- 自定义 Hook 封装可复用逻辑，文件命名 `use<Name>.ts`

### 4.3 Zustand 状态管理模式

```typescript
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

interface ChatState {
  sessions: ChatSession[];
  activeSessionId: string | null;
  isLoading: boolean;

  // 动作方法
  loadSessions: () => Promise<void>;
  setActiveSession: (id: string | null) => void;
}

export const useChatStore = create<ChatState>()(
  persist(                              // 需要持久化的 store 使用 persist
    (set, get) => ({
      sessions: [],
      activeSessionId: null,
      isLoading: false,

      loadSessions: async () => {
        set({ isLoading: true });
        try {
          const data = await chatApi.getSessions();
          set({ sessions: data, isLoading: false });
        } catch {
          set({ isLoading: false });
        }
      },
      setActiveSession: (id) => set({ activeSessionId: id }),
    }),
    {
      name: 'chat-storage',
      partialize: (state) => ({
        sessions: state.sessions,       // 只持久化数据字段
      }),
    },
  ),
);
```

**规则：**
- 命名：`use<Name>Store`
- 文件：`<name>.store.ts`
- 类型完整声明，包含 state 和 actions
- Actions 直接定义在 create 回调中
- 跨 store 访问使用 `useAuthStore.getState()`
- 避免循环依赖（使用动态 `import()` 或 getState）

### 4.4 API 调用层

```typescript
// api/client.ts
const apiClient = axios.create({
  baseURL: import.meta.env.VITE_API_URL || '/api/v1',
  timeout: 30000,
});

// 请求拦截器：附加 Token
apiClient.interceptors.request.use((config) => {
  const token = useAuthStore.getState().accessToken;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// 响应拦截器：401 自动刷新 Token
apiClient.interceptors.response.use(
  (response) => response.data,
  async (error) => {
    if (error.response?.status === 401) {
      // 自动刷新 Token 并重试
    }
    return Promise.reject(error);
  },
);
```

**规则：**
- `apiClient`：带认证拦截器的实例（大部分接口使用）
- `noAuthClient`：无认证实例（仅登录/注册/刷新使用）
- API 方法按模块分组：`authApi`、`chatApi`、`agentApi`、`knowledgeApi`、`userApi`
- 响应拦截器自动解包 `response.data`，调用方直接获取数据

## 5. 测试规范

### 5.1 后端测试（Jest）

```typescript
import { Test, TestingModule } from '@nestjs/testing';

describe('AuthService', () => {
  let service: AuthService;

```

**规则：**

- `afterEach` 调用 `jest.clearAllMocks()`
- Mock 所有外部依赖（Prisma、Redis、外部 API）
- 测试覆盖正常路径 + 边界情况 + 异常路径

## 6. AI Agent 规范

### 6.1 意图分类约定

AgentOrchestrator 支持三种模式，由 IntentClassifier 路由：

| 意图 | 路由策略 | 适用场景 |
|------|---------|---------|
| `SIMPLE` | Direct LLM (DeepSeek V3) | 简单问答，无需工具 |
| `TOOL_USE` | ReAct Engine | 需调用工具（搜索、计算等） |
| `COMPLEX` | Plan-and-Execute | 多步骤复杂任务 |
| `REASONING` | DeepSeek R1 | 深度推理问题 |
| `CREATIVE` | Direct LLM (DeepSeek V3) | 创意生成任务 |

### 6.2 工具注册标准

```typescript
// 工具定义接口
interface ToolDefinition {
  name: string;                              // 工具名：snake_case
  description: string;                       // 描述：LLM 理解用途
  parameters: Record<string, {               // 参数描述
    type: string;
    description: string;
    required?: boolean;
  }>;
  requiresSessionMembership?: boolean;       // 是否需要会话成员权限
  handler: (args: any, ctx: AgentContext) => Promise<any>;  // 执行函数
}
```

**规则：**
- 工具名使用 `snake_case`
- 描述清晰说明工具的用途和适用场景
- 参数标明类型、描述、是否必需
- handler 返回结构化的结果数据
- 新工具在 `ToolRegistry` 的 `registerBuiltInTools` 中注册

### 6.3 RAG 知识库集成

- 文档上传后通过 `RagEngine.chunkAndStore` 分块 + 向量化
- Agent 通过 `search_knowledge_base` 工具自动检索用户知识库
- 向量搜索不可用时自动降级为关键词搜索 (`ILIKE`)
