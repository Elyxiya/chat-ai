开发流程规范 v1\.0

## 一、规范概述

### 1\.1 技能名称

轻量级研发流程

### 1\.2 核心描述

本规范定义团队标准化轻量级研发流程，完整链路为：**需求 → 验收标准（AC）→ 技术方案 → 关键测试用例 → TDD 编码 → 评审与文档同步**。适用于从需求讨论到可交付代码的全链路开发任务，强制约束研发全流程动作，保障业务代码可验证、可测试、可维护，统一团队研发规范，规避随意开发、测试缺失、文档滞后等问题。

### 1\.3 兼容范围

兼容所有支持 Markdown 指令的 AI 编程助手，包含 GitHub Copilot、Cursor、CodeGPT 等工具。

## 二、适用场景

- 用户提出新功能开发、线上问题修复、代码重构类研发需求

- 需要严格遵循团队规范，标准化落地的各类开发任务

- 需要辅助生成验收标准、技术方案、测试用例、TDD 代码的研发场景

## 三、核心原则（强制遵守）

- **无 AC 不开发**：每条需求必须配备唯一、可验证、无歧义的 Given\-When\-Then 格式验收标准，无验收标准禁止进入开发环节。

- **TDD 先行原则**：严格遵循先写失败测试、后编写业务代码的「红\-绿\-重构」开发逻辑。

- **全场景测试覆盖**：所有核心功能必须覆盖快乐路径、边界值、异常路径、并发/幂等四大测试场景。

- **文档同步闭环**：所有代码变更对应的文档必须与 PR 同步更新，未完成文档更新禁止合并代码。

## 四、标准化执行流程

### Step 0：确认任务上下文（启动前置）

#### 执行动作

开发启动前必须明确完整任务信息，缺失信息主动追问补齐，核心确认内容：

- 详细需求描述

- 预期交付物标准

- 涉及研发模块（前端/后端/数据库/API/中间件等）

#### 准入要求

上下文信息不完整，禁止进入后续研发流程。

### Step 1：需求梳理 \&amp; 输出验收标准（AC）

#### 执行动作

1. 将原始需求拆解为 1\~N 条独立的用户故事/功能点；

2. 每条功能点必须输出 **Given\-When\-Then** 标准化验收标准；

3. 若需求方无法提供 AC，自动生成候选 AC 清单，待用户确认后落地。

#### AC 标准模板

```gherkin
Given [前置状态]
When [用户操作或系统事件]
Then [可观测、可验证的结果]
```

#### AC 示例

```gherkin
Given 用户已登录且购物车有有效商品
When 用户点击「去结算」按钮
Then 系统展示订单确认页，完整展示商品列表、合计总价等信息
```

#### 合规检查点

所有 AC 必须满足：独立无耦合、结果可量化验证、不依赖具体代码实现细节。

### Step 2：轻量级技术方案设计

#### 执行时限

单任务方案设计时长 ≤30 分钟，轻量化落地，避免过度设计。

#### 输出内容（至少包含一项，复杂场景多项输出）

- **核心接口定义**：入参、出参、字段说明、自定义错误码

- **数据库变更**：新增/修改字段、索引调整、数据迁移脚本

- **异常处理策略**：超时、重试、降级、事务补偿、兜底方案

- **服务时序图**：多服务、多模块交互场景必须输出时序图

#### 输出规范

统一使用 Markdown 格式输出，可直接作为 PR 描述、团队 Wiki 存档内容。

#### 核心校验规则

方案落地校验：**基于当前方案可直接编写失败测试用例**，若无法实现则重新迭代方案，直至合规。

#### 最小技术方案模板

```markdown
## 接口
- 请求方式&地址：POST /api/xxx
- Request：{参数名: 参数类型, 说明}
- Response：{返回字段: 类型, 说明}
- 错误码：自定义业务错误码及含义

## 数据库
- 表变更：xxx 表新增/修改字段 xxx（类型、默认值、约束）
- 索引变更：新增/删除索引（字段、索引类型）

## 异常处理
- 场景1：异常原因 → 处理逻辑、返回结果
- 场景2：超时/重试策略、降级兜底方案
```

### Step 3：编码前编写关键测试用例

#### 核心要求

基于 AC 验收标准与技术方案，全覆盖四大测试场景，**先写测试、后写代码**，禁止直接编写业务实现代码。

#### 测试用例配比规范

- 快乐路径：40%（核心正常业务场景）

- 边界值场景：20%（参数临界、数值极值、数量上限/下限）

- 异常路径：30%（报错、超时、参数非法、资源不足）

- 并发/幂等场景：10%（接口重复请求、并发操作、数据一致性场景，按需启用）

#### 标准化测试用例表格模板

|测试类型|场景描述|预期结果|
|---|---|---|
|快乐路径|正常业务流程，参数合法、资源充足|业务执行成功，返回正常结果，数据状态正确|
|边界值|参数为临界极值、数量达上下限|按业务规则正常校验，返回合规结果/合法报错|
|异常路径|数据库超时、参数非法、资源不足|事务回滚/拦截请求，返回对应错误码，无脏数据|
|并发/幂等|同一请求重复提交、多线程并发操作|业务幂等执行，无重复操作，数据一致|

#### 辅助能力

可根据技术栈生成对应测试代码框架，支持 JUnit、pytest、Jest 等主流测试框架。

### Step 4：TDD 红\-绿\-重构编码

#### 循环执行逻辑（最小粒度迭代）

1. **红（失败测试）**：针对单条 AC、单个场景，编写最小粒度的失败测试用例；

2. **绿（代码通过）**：编写最少、最简的业务代码，使当前失败测试通过，不提前做过度优化；

3. **重构（优化迭代）**：优化代码结构、规范代码风格，保证所有已有测试全部通过。

循环迭代直至所有核心场景、所有 AC 对应的测试用例全部通过。

#### 代码提交前置强制校验

- 本地测试 \+ CI 流水线测试全部通过，无失败用例；

- 整体代码测试覆盖率 ≥60%，核心业务逻辑覆盖率 ≥80%；

- 禁止使用 @SuppressWarnings、硬编码等方式绕过测试报错与校验。

### Step 5：代码评审与测试验证

#### 标准化 PR 描述模板（强制包含）

- 本次迭代关联的全部 AC 验收标准列表

- 四大场景测试用例覆盖摘要及通过率

- 技术方案简述或 Wiki 文档链接

- 文档更新自检勾选项

#### 核心评审重点

- 代码是否存在空 catch 块、无日志静默失败风险；

- 异常场景、边界场景是否全部覆盖对应测试用例；

- 代码可读性、结构合理性、可维护性；

- 业务逻辑是否完全匹配 AC 验收标准。

#### 合并硬性条件

CI 流水线全部绿灯通过 \+ 至少 1 人代码 Approve，缺一不可。

### Step 6：文档同步更新（PR 强制闭环）

#### 差异化文档更新规则

- **API 变更**：同步更新 Swagger 文档、接口注释、调用说明；

- **架构/方案变更**：在 /docs/adr/ 目录下新增 ADR 架构决策文档；

- **配置/部署变更**：同步更新 README、运维操作手册、部署说明。

#### 强制校验规则

PR 描述必须勾选「文档已同步更新」，未完成文档更新**禁止合并代码**。

#### ADR 架构决策文档模板

```markdown
# 文档标题：[架构/方案变更主题]

## 背景
描述当前存在的问题、业务痛点、改造必要性。

## 决策
明确本次采用的技术方案、架构调整、核心实现思路。

## 后果
- 正向收益：性能提升、风险规避、流程优化等
- 引入成本：新增依赖、运维成本、需要兼容的问题等

```

## 五、角色与职责

|角色|核心职责|
|---|---|
|需求方|提供原始需求，确认 AC 验收标准，验收最终交付效果|
|开发|方案设计、编写测试用例、TDD 编码、同步更新文档、提交 PR|
|测试|补充手工测试、探索性测试，评审测试覆盖完整性，校验业务合规性|

**协作默认规则**：AI 助手默认承担「开发」角色，用户可兼任需求方、测试角色。

## 六、违规行为拦截规则（强制生效）

|违规行为|标准拦截响应|
|---|---|
|未提供 AC 直接要求编写代码|拒绝开发，提示：「请先提供 Given\-When\-Then 格式验收标准，无 AC 不开发」|
|测试用例仅覆盖快乐路径，缺失异常/边界场景|强制拦截，提示补充至少 3 个异常、边界类测试用例，完善覆盖度|
|无技术方案，直接启动编码|禁止编码，要求先输出接口签名、数据库变更等最小技术方案|
|PR 未同步更新文档，申请合并|阻止合并，触发文档生成辅助能力，强制补齐文档更新|
