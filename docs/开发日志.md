### 1. Embedding 服务抽象（替换 DeepSeek Embedding）

**根因**: `api.deepseek.com` 不提供 embedding API 端点（`/v1/embeddings` 返回 404），需要更换 embedding 提供商。

**修复**:
- 创建 [embedding.service.ts](../apps/api/src/modules/llm/providers/embedding.service.ts) — 可注入的 EmbeddingService，支持多提供商：
  - `openai`: 使用 `text-embedding-3-small`（1536 维，兼容当前数据库 schema）
  - `deepseek`: 作为兼容选项保留（会优雅地返回空数组）
- 通过 `.env` 的 `EMBEDDING_PROVIDER=openai|deepseek` 切换
- RAG 引擎 (`rag-engine.service.ts`) 注入 EmbeddingService，不再直接依赖 DeepSeekProvider
- ToolRegistry 也注入 EmbeddingService 并传递给 RagEngine

**环境变量新增**:
```
EMBEDDING_PROVIDER=openai
OPENAI_API_KEY=
OPENAI_EMBEDDING_URL=https://api.openai.com/v1/embeddings
OPENAI_EMBEDDING_MODEL=text-embedding-3-small
```

### 2. Redis LRANGE 参数兼容性修复

**根因**: ioredis v5 与 Redis 服务器之间可能出现 `LRANGE` 命令参数序列化兼容性问题，报 `ERR value is not an integer or out of range`。MemoryService 虽然已有 try-catch 保护，但日志噪音较大。

**修复**:
- `RedisService.lrange` 对 `start`/`stop` 参数使用 `Math.floor()` 确保整数，同时改为捕获异常返回 `[]`，不再向上抛
- `RedisService.ltrim`、`RedisService.expire`、`RedisService.set` 同样使用 `Math.floor()` 防御
- MemoryService 中 `getShortTermMemory` 添加参数有效性校验（`Number.isFinite`），默认值兜底

**变更文件**:
- [apps/api/src/modules/common/redis.service.ts](../apps/api/src/modules/common/redis.service.ts)
- [apps/api/src/modules/agent/memory/memory.service.ts](../apps/api/src/modules/agent/memory/memory.service.ts)

### 4. DeepSeek 思考模式参数修复

**根因**: `reasoning_effort` 错误地放在了 `extra_body` 内部，但 DeepSeek 官方 API 要求 `reasoning_effort` 是顶级请求参数，只有 `thinking` 开关放在 `extra_body` 中。

**修复**:
- `buildExtraBody()` 仅保留 `thinking: { type: 'enabled' }`
- `reasoning_effort` 移到 `body` 顶级，与 `model`、`messages` 同级
- 同时在 `chat()` 和 `chatStreamWithReasoning()` 中修正

**变更文件**:
- [apps/api/src/modules/llm/providers/deepseek.provider.ts](../apps/api/src/modules/llm/providers/deepseek.provider.ts)
